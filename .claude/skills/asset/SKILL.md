# asset

Replace a `playable` game's deliberate **primitive** visuals with real, coherent **SVG art**, so the title goes from "intentional toy" to "looks designed" — with the re-skin legibly recorded in the manifest. The real deliverable is a sharp re-skin **system**, not the prettier game: every gap must be attributable to specific prose here. This runs as a clean bolt-on **after `playable`**:

```
concept → builder → validator → [playable] → asset → validator(re-run) → [styled]
```

## Choosing the method (branch on `concept.art_direction`)

Before choosing a method, **read `concept.theme`** (the title's modality-neutral world — premise/tone/mood_keywords/setting). `art_direction` is the **visual expression of `concept.theme`**: the visual system you derive below must read as *that theme's world* — the same premise/tone/setting the audio and the store icon also express — not a free-standing aesthetic. Honor the theme; do not reinterpret it. (Reading the theme is not editing it — the "consume `concept` as-is" rule still holds.)

This skill has **two methods**, picked from `art_direction`. Both share *all* the rewiring craft below (texture-on-disk → `Sprite2D`/`TextureRect`/`draw_texture*`, footprint placement, primitive removal, the run-007 immediate-mode lessons) — only how the texture is *produced* differs.

- **geometric / neon / flat / hyper-casual / UI** → **`svg`** method (authored inline as text; the rest of this skill below). Resolution-independent, covers every density bucket from one file.
- **representational / character / creature / illustrated / textured** → **`raster`** method (generated RGBA PNGs via local ComfyUI + SDXL + LayerDiffuse; see **The `raster` method** section). This is the art SVG does *badly* — a painted hero, a creature, a textured surface.

**Raster is the default for art; do not retreat to SVG to dodge a quality problem** (owner directive 2026-06-02). SVG is reserved for a genuinely good reason — a pure UI/HUD/geometric element where vector resolution-independence is a real win. The remedy for "terrible" raster is to **lift raster quality** (backgrounds, sizing, the tuning levers in the raster method below), not to fall back to vectors. State the justification when you do choose `svg` for an entity.

This is the same "SVG aesthetic boundary" the SVG method documents — `raster` is now where the skill **goes** when it hits a representational concept, instead of only flagging it. A single run may be **mixed-method** (some entities `raster`, some `svg`, some left `primitive`); see the raster section's "Mixed-method honesty".

## Inputs
- `manifests/<id>.json` with a populated `concept` block and `status = "playable"`.
- The generated project on disk at `games/<id>/` (a single `Node2D` whose `Main.gd` draws every entity procedurally in `_draw()`).
- The pinned Godot version from `README.md`.

## Outputs
- `games/<id>/art/*.svg` — one Claude-authored vector file per re-skinned entity.
- A rewired `Main.gd` / `Main.tscn` displaying those SVGs via `Sprite2D` (world) / `TextureRect` (HUD).
- A populated `asset_pass` block and flipped `assets[]` entries (`origin:"svg"`).
- `status = "styled"` (after the validator re-run passes).

## Hard requirements
- The re-skinned project MUST still import and run **headless with no script errors** (the `validator` re-enforces this).
- Game **logic is untouched** — movement, collision, spawning, scoring, input, and game-over/restart behave exactly as before. Only the *visual representation* changes.
- No double-draw: for every re-skinned entity the old `_draw()` primitive is removed or guarded. Leaving the primitive *and* adding the sprite is the most common failure — prevent it.
- No new MCP tool and no new dependency: author SVG inline as text, exactly as `builder` authors GDScript. Godot's native importer rasterizes it.
- Do **not** edit `concept` or `builder`. Consume `concept.art_direction` as-is.

## Step 0 — Derive the visual system FIRST (the real deliverable)

A real asset creator does not produce a pile of independently-acceptable shapes — it produces **one coherent visual system** and applies it everywhere, so the game reads as a single designed thing. "Each SVG is fine but they don't cohere" is the **primary failure** this step exists to prevent (the art analog of the M0 hybrid finding, "two systems that coexist instead of fuse"). When it happens, it is attributable to *this system*, not the individual files.

Before authoring **any** SVG, write the system down explicitly from `concept.art_direction`:

- **World/character bible (derive this FIRST)** — from `concept.theme.setting` + `premise`, pin **one fictional world and one character family** before any subject or silhouette is chosen. Every actor — hero, hazard, pickup — is an *inhabitant of that one world*, drawn from that one family (same materials, same era/genre, same "what kind of thing is this"). The hero and the hazard must read as belonging together — **not a cyan robot vs a red ogre** (the crosser-0001 finding). This is the *subject-level* sibling of the palette/form/shading below, which pin only *how* things are rendered, never *what world they are from*; that missing sibling is what let two individually-clean sprites read as different worlds. Recorded verbatim in `asset_pass.visual_system.world_bible`. Applies to **both** methods.
- **Palette** — a fixed 3–5 colour set with named roles (primary, accent, danger, background, …). Every fill/stroke comes from this set.
- **Line/stroke** — one stroke weight and one join/cap style, used throughout.
- **Form language** — corner radius, geometric vs. organic, level of detail. Pick one and hold it. Then give **each** actor *one* signature silhouette detail (a directional notch on the player, an inner facet on a pickup) — staying within the form language **and within the one character family from the world bible** (a variation on the shared family, never an unrelated shape). Without it the re-skin reads as "the same primitive, just rounder" and the upgrade is muted (run-007 finding #3): a re-skinned square that is still just a square is a weak A/B.
- **Shading model** — flat / single-direction gradient / glow halo — pick **one** and apply it to every asset.
- **Scale & padding** — how each SVG maps to its primitive's footprint, plus consistent internal padding so assets sit together (e.g. all art drawn into a square `viewBox` with the same % padding).

This system is recorded verbatim in `asset_pass.visual_system` (below) so it is reviewable, reusable, and the thing a run report critiques when the result looks incoherent.

## The SVG aesthetic boundary (be honest about scope)

SVG is the *right* tool for the art this pipeline produces today — abstract, geometric, neon/flat, hyper-casual, all UI. Resolution independence is an app-store strength: one vector covers every Android density bucket. SVG is the *wrong* tool for **representational, character, illustrated, or textured** art (a painted hero, a photoreal background) — that is an **M1.5 (raster / local-SD)** concern.

If a concept's `art_direction` leans representational enough that hand-authored vector would be weak, **say so in `asset_pass.notes`** ("art_direction calls for an illustrated character; SVG would be mediocre here — M1.5 would serve it better") rather than silently shipping poor vector art. Re-skin what SVG does well; flag what it doesn't.

## Authoring the SVGs

One file per builder-registered visual entity, under `games/<id>/art/` (e.g. `player.svg`, `obstacle.svg`, `pickup.svg`). Every file conforms to the Step 0 system: same palette, same stroke, same form language, same shading model, same `viewBox`/padding convention.

- Use a square `viewBox` (e.g. `0 0 100 100`) so scaling to a footprint is predictable.
- Execute the shading model — if it is "flat fill + outer glow halo", actually draw the halo (an oversized, low-alpha shape behind, or an SVG `<filter>` blur), don't just assert it. Asserting neon is not executing neon.
- Keep files small and diffable — paths, `<rect>`, `<circle>`, gradients, filters. No embedded rasters.

## The SVG-swap mechanism (the technically hard part)

The builder draws procedurally — there are **no sprite slots to swap into**, and `Main.tscn` is a bare `Node2D` with all rendering in `Main.gd`'s `_draw()`. So you must both generate SVGs *and* rewire the game.

**a) Get the SVGs into Godot.** Godot 4.x imports `.svg` as a texture via a `.svg.import` sidecar (carrying a `scale` param). Run a headless import pass so the sidecars and cached textures exist **before** re-validating:
```
godot --headless --path games/<id>/ --import
```
Commit the generated `*.svg.import` sidecars alongside the art (expected Godot output, like `.gd.uid`).

**b) Replace each primitive (the documented swap pattern).** Per re-skinned entity:
- Load the texture and display it with a node positioned/scaled to the primitive's original footprint:
  - **World actors** (player, obstacles, pickups) → `Sprite2D` with `texture = load("res://art/<name>.svg")`, placed at the same world transform the `_draw()` used. For pooled/spawned collections (e.g. obstacles), create one `Sprite2D` per live instance and move it each frame to the position the primitive was drawn at, instead of a `draw_*` call.
  - **HUD/UI** → `TextureRect` under the HUD layer.
- **Remove or guard** the matching `_draw()` code for that entity — delete its `draw_rect`/`draw_circle`/glow calls; keep the node's transform and all game logic. Verify you did not leave the primitive drawing underneath the sprite (double-draw).
- Movement, collision, spawning, scoring — **unchanged**.

**c) What stays primitive.** Effects (glow halos, particles, screen-shake, squash/stretch, flash) stay code — they are *motion/juice*, not art. Backgrounds may stay procedural (parallax grid/stars) **or** get a tiling SVG — your judgment from `art_direction`. **Record which entities you re-skinned vs. left primitive** so a partial re-skin is a legible choice, not a silent gap. **A background left primitive is a cohesion gap, not a free pass:** sprites floating on untextured primitive bands undercut the one-rendered-system goal. As of round-3 the raster method **generates a themed background** (see "Backgrounds & composition") — that is now the expected fix. If you nonetheless leave the background primitive, you **must** list it in `left_primitive` and justify it in `asset_pass.notes`; never silently omit it.

**d) Re-skinning an immediate-mode `_draw()` game (the builder's DEFAULT — read this first).** The builder ships a *single* `Node2D` whose `Main.gd` draws every entity in one root `_draw()`, with **zero child nodes** and screen-shake applied via `draw_set_transform`. Two swap strategies, and the choice is itself a finding to record in `asset_pass.notes` (run-007 findings #1–#2):

- **Lighter, preferred for immediate-mode games — `draw_texture*` in place.** Replace each actor's `draw_rect`/`draw_circle` with `draw_texture_rect()` / `draw_texture()` *inside the existing `_draw()`*, loading the SVG texture once in `_ready()`. This reuses the SVG art while preserving z-order, the shake transform, and squash/stretch **for free** — no node tree surgery. Reach for this first when the game is one `_draw()`.
- **Retained `Sprite2D`/`TextureRect` nodes — for node-based scenes, or when you want per-actor nodes.** If you go this route on an immediate-mode game, expect three non-obvious steps the naive "just add a Sprite2D" misses:
  1. A parent's own `_draw()` renders *below* its children. So keep the **background layer** in the root `_draw()`, but move anything that must sit **above** the actors (HUD, particles, crash-flash) into a **higher-`z_index` child node** that delegates back (e.g. an `Overlay` Node2D whose `_draw()` calls a `draw_overlay(self)` method on Main). Otherwise the HUD vanishes under the sprites.
  2. **Hoist screen-shake into shared per-frame state** (compute the shake offset once in `_process`, store it, add it to every sprite's `position` *and* use it for the background `draw_set_transform`) so sprites and background shake together. A sprite's node transform does not inherit the `_draw()` transform.
  3. **Pool one node per spawned actor** (grow an `Array` of `Sprite2D`, show/hide per live instance). For a tinted family (e.g. obstacles in two danger colours from one white SVG), set `modulate` per instance.

**Failure attribution (the POC value):** a bad re-skin is always attributable — you authored a poor SVG, mis-positioned/mis-scaled a sprite, or failed to remove the underlying primitive. Each is a specific, fixable prose gap.

## The `raster` method (representational art via local SD)

Use this when the method branch sent you to `raster`. It produces **RGBA sprites** (native transparency at generation time) that hand-authored SVG cannot do well. The deliverable is still a sharp **system**: every "this sprite looks bad" must trace to a fixable cause — a weak prompt scaffold, a wrong style profile/param, a mis-placed sprite, a left-in primitive, or an *infra* failure in `comfy.mjs` — never an unattributable blob.

### Prerequisite (assumed running, like Godot)
ComfyUI runs headless as a local server (default `http://127.0.0.1:8188`) with an SDXL checkpoint and the **ComfyUI-layerdiffuse** node installed. It is **not** managed by this skill — the owner starts it. Before generating, confirm it is reachable and see what checkpoints exist:
```
node tools/comfy.mjs --check
```
Expected: `comfy OK at http://127.0.0.1:8188 — N checkpoint(s): ...`. If it prints `UNREACHABLE`, **stop** and ask the owner to start ComfyUI (or `! <launch command>`). A failure here is *infra*, attributable to the stack, not to your art judgment — never work around it by faking a PNG.

**Stack-version sensitivity (hard-won — see `docs/superpowers/m1.5-feasibility-notes.md`).** ComfyUI-layerdiffuse is **version-fragile**: on a bleeding-edge ComfyUI it (a) silently drops its model patch (`patch type not recognized` in the server log → output is haze) and (b) errors on the final alpha-join. The feasibility gate pinned **ComfyUI to a node-compatible release** + a small local node patch to get clean transparency. If sprites come back as haze or with a desaturated/gray interior (correct alpha but flat color), that is a **stack-version** issue, not your recipe — check the server log for patch warnings and confirm the pin/patch from the feasibility notes are in place before touching the recipe.

### Step 0 (raster) — visual system first, with style as a first-class choice
Do the normal Step 0 (palette, form, shading, scale). For raster, additionally:

**(a) Select the per-game style profile — explicitly.** Art style is a **first-class, per-game parameter**; it is what lets *one* skill make *different-looking* games. From `art_direction`, choose:
- a **checkpoint** (e.g. a painterly SDXL finetune vs. a flat-cartoon vs. a pixel-art checkpoint),
- optional **LoRA(s)**,
- the **style fragment** of the prompt (e.g. `"painterly, illustrated, soft brushwork"`).

Justify "this `art_direction` → this profile" exactly as you justify the SVG visual system. Record it verbatim in `asset_pass.visual_system.style` (`{ checkpoint, loras[], style_prompt }`). Two different games **should** look deliberately different because they picked different profiles — that difference is the goal, not a failure.

**(b) Fix the shared prompt scaffold.** Write one base prompt string (style fragment + shared scene/lighting/background terms) and one fixed sampler/steps/cfg. **Every** sprite in this game is generated as `scaffold + this actor's subject` from that one scaffold, so the set reads as a single designed family. Record it in `asset_pass.visual_system.prompt_scaffold`. Always include the phrase `"single centered subject, full body, clean transparent background"` in the shared scaffold (or every per-actor subject), and put multiplicity/scene terms — `"scene, multiple characters, frame, border, ring, wreath, ground, floor, shadow"` — in the shared `negative`. Without this, an over-elaborated actor (a small pickup with a decorative description) often comes back as a leaf/wreath/row-of-blobs rather than one isolated subject; the negatives don't fire if the positive prompt itself invites a scene.

**"Each sprite is fine but they don't cohere"** is the same *primary failure* the SVG method guards against — here prevented by the **world bible** + profile + shared scaffold + fixed params, **not** by independent per-actor prompts. There are two kinds of incoherence: *style* divergence (caught by the profile/scaffold/params) and **subject-world divergence** (caught by the world bible) — a cyan-robot hero next to a red-ogre hazard is the latter, a finding about the **world bible**, never about an individual PNG.

### Audit every generation against intent — subject, tone AND cross-asset style (2026-06-02, owner directive)
A diffusion model does **not** reliably draw what you asked. **Read every generated PNG and judge it against intent before accepting it** — on *three* axes, and regenerate (don't ship "close enough"):
- **Subject fidelity** — is this the *thing* you asked for? (Probe findings: "a small impish fire-demon …" rendered a tall **winged demoness**; "forked arcs of lightning chaining …" rendered a **canyon landscape with one bolt**.) When it drifts, **tighten the subject phrase** — make the intended thing the *explicit sole subject*, add concrete shape/scale words ("small, squat, hunched, oversized head"), and add **anti-drift negatives** for what it became (`woman, female, elegant, glamorous, tall` for the demoness; `landscape, scenery, canyon, terrain, horizon, ground` for the scene drift).
  - **A NAMED card must depict the thing it names — recognizably, at a glance. This is gameplay legibility, not just style.** A card called "Chain Lightning" has to read as *chained forked lightning* (multiple bolts branching/arcing between points); "Fireball" as a hurled ball of fire; "Frost Shard" as ice spikes. The proven failure (2026-06-02 probe): abstracting the spell into a generic "arcane sigil / rune emblem" produced a pretty **ornamental medallion with crystals** that had *no lightning in it at all* — the player can't tell what the card does from its art. **Do not let "iconic spell illustration / sigil / emblem" abstract away the spell's literal, identifiable phenomenon.** Name the actual effect as the sole subject ("multiple forked violet-white lightning bolts branching and arcing, crackling electric energy"), and **negative out the generic-icon attractors** (`medallion, emblem, sigil, rune circle, coin, badge, gemstone, crystal, ornamental icon, symmetrical`) when they hijack the spell. Audit each card by asking "could a player name this spell from the art?" — not just "is it on-style?"
- **Tone fidelity** — does it match the title's *mood*? Subject-right is not enough: a corrected "small imp" can come back **cute/chibi** when the world wants sinister. Pull tone with mood words in the positive and the wrong tone in the negative (`cute, adorable, chibi, mascot, friendly` to banish; `menacing, sinister, grotesque, fanged` to invite). Tone is part of `concept.theme` — judge against it, not in the abstract.
- **Cross-asset style cohesion — judge each asset *against the others*, not only in isolation.** A shared style fragment is **necessary but not sufficient**: the *same* words land in different rendering registers depending on subject. The proven failure (2026-06-02 probe): one fragment rendered an abstract **spell-effect card** as a glowing, painterly, atmospheric illustration — but rendered a **creature** from the same fragment as semi-**realistic** (rendered musculature, photoreal anatomy/skin). Side by side they don't cohere even though palette + words matched ("the imp is way too realistic vs. the card"). **Diffusion pulls representational subjects (creatures, characters) toward photo-realism harder than abstract subjects (effects, sigils).** So: (a) put the assets next to each other and ask "do these read as one illustrator's hand?" — not just "is each one fine?"; (b) pick the asset whose style the owner approved as the **reference**, and **conform the others to *that specific register*** — don't conform to an abstract idea of "stylized."
  - **De-realism ≠ flatten. This is the trap that bit this exact probe (a second, owner-flagged miss).** When the realistic creature was "corrected," the easy over-swing is to pile on `cel-shaded, flat color areas, thick clean outlines, bold graphic shapes` — which makes a **flat vector mascot**, and if the reference card is actually *painterly and luminous*, the creature is now incoherent in the **opposite** direction ("the imp looks flat while the card looks detailed and whimsical"). The defect in the realistic creature was specifically **photoreal anatomy/skin**, *not* that it was painted. So remove **only** the photoreal-anatomy attractor (`photorealistic, hyperrealistic, realistic human skin, detailed realistic musculature, photo, 3d render, octane render`) and **keep the reference's painterly qualities** (`hand-painted digital illustration, painterly rendering, rich dramatic shading, soft volumetric lighting, glowing arcane aura, magical luminosity, whimsical`). Match the reference's *level of rendered detail and glow*, not a generic flatness. If the reference were flat-graphic you'd flatten; because it's painterly-glowy you keep paint and only de-photo-realize.
  - This is the cross-asset analog of the world-bible's subject-cohesion check (above): style divergence across assets is a finding about the *shared profile's interaction with subject type*, attributable to this system — fix it by conforming to the approved reference's actual register (paint vs. flat, glow level, detail level), not by reaching for whichever knob is easiest.
  - **"Same medium" is NOT cohesion — scrutinize finish, line, detail and mood, and watch for whack-a-mole.** "Both are 2D" (or both painterly, both the right palette) is a *single shared attribute*, and it will lull you into declaring a match while the assets still read as two different artists. Apply the one-hand test on the **specific sub-axes**: (1) **finish/render level** — flat cartoon-sticker vs. ornate rendered vs. painterly-soft; (2) **line weight** — bold cartoon contour vs. fine ornamental linework vs. no outline; (3) **detail density** — clean-minimal vs. busy-ornamental; (4) **mood** — cute/mischievous vs. menacing vs. solemn. The proven trap (2026-06-02 probe, multiple rounds): each regen fixed *one* axis and slipped on *another* — first the creature was too 3D-rendered next to a flat card; corrected, the creature went flat-cute while the card became an *ornate metallic medallion*; so they never cohered even though every round shared *some* attribute. The fix is not to keep swapping which asset is "too much" — it is to **lock ONE finish register up front** (pick the reference the owner approved, name its finish/line/detail explicitly), then **conform every asset to that same register at once** and judge a true side-by-side: "would one illustrator have drawn *both* of these?" Iterate against that question, not against any single shared trait. Generate the actual probe **pair under one locked fragment** (regenerate the reference asset too, so you're comparing the locked style to itself), never an old asset against a new one.
- **The "iconic illustration / dramatic composition / backdrop" card scaffold biases toward scenic landscapes.** For a *spell-effect* card, say the effect is the **sole subject** ("the lightning spell effect is the sole subject, no ground") and put landscape terms in the negative — otherwise SDXL paints a vista the effect merely decorates.
- Same lever order as a fighting-the-medium actor: subject/tone phrase → negatives → `master_resolution`/`width,height` → `cfg ±1`. Keep the *style* fragment + params fixed (cohesion); only the subject/tone/negatives move per regen.
- **The audit extends to the COMPOSITED render, not just the raw PNG** — when code draws chrome over the art (full-bleed cards, HUD over a background), eyeball the assembled widget for layout collisions/clipping too (see "Text & chrome over full-bleed art"). "The art is good" ≠ "the screen is good."

### Audit the COMPOSITED, RUNNING game for MISSING assets + visual bugs — generating the hero actors is not the whole pass (2026-06-03, owner directive)
Making the obvious actors (enemies, cards, background) and wiring them is **not** a finished art pass. After wiring, **screenshot the actual running game and inventory EVERY drawn element**, classifying each as `art` / `code-chrome-by-design` / `primitive-that-should-be-art`. The owner's finding: a re-skin that nails enemies+cards+bg still looks unfinished because the *surrounding UI* is all primitives. Walk the renderer's `_draw()` (or scene tree) top to bottom — most of these start life as primitives and are easy to forget:
- **Card backs + draw/discard piles** — often just a text count; a deckbuilder reads unfinished without a card-back image and a visible pile stack.
- **Player HUD frame + portrait/avatar, mana orbs/crystals, HP-bar frames** — primitive bars/circles read cheap next to painted cards.
- **Buttons** (End Turn, Skip, confirm) — primitive rects.
- **Status-effect icons** (burn/chill/poison…), **intent indicators** (attack/defend/buff shown as *text* vs an icon), **block/shield badges** — primitives that carry gameplay meaning and benefit most from clear iconography.
- **Relic / passive / artifact icons, currency, map/node icons** (combat/elite/rest/shop/boss), **reward & win/lose screens**.
Then scan the same screenshot for **visual bugs**: art **squished** by an aspect mismatch (e.g. a 1280×768 bg in a 1280×720 frame — generate backgrounds at the *exact* viewport aspect), a sprite **not grounded** to its floor/anchor (transparent-padded square sprites float unless you bottom-anchor the draw rect), HUD text/bars **overlapping or clipping** (esp. over a busy background with no panel behind them), illegible chrome at true on-screen size, **z-order** mistakes, and **double-draws** (a leftover primitive showing through new art). Produce a concrete **missing-asset list + bug list** and surface it to the owner with a scope decision — "I generated the heroes" ≠ "the art pass is done." This is the art analog of the completeness-critic pass.

### Every UI element must be FINISHED, COHESIVE, LEGIBLE, and detail-matched to its on-screen size (2026-06-03, owner directive)
The inventory audit above flags *missing* assets; this one flags *under-finished* ones — and it is where a re-skin most often still reads "unfinished" after the hero art is great. **Code-drawn primitives — HP/resource bars, buttons, mana/energy orbs, status pips, rarity/cost gems, intent icons — are part of the visual system, NOT exempt from it.** A flat default `draw_rect`/`draw_circle` reads as unstyled programmer-art next to painted assets and drags the whole screen down. For EACH such element, the audit must check four things:
- **Finished, not placeholder.** Does it look intentionally designed, or like a half-drawn glyph? (Owner findings on this game: the mana orbs and the burn/fire debuff icon read "incomplete"; the small element/rarity pip reads like an unfinished gem.) A bare circle/rect is a placeholder — give it a deliberate shape, fill, outline, and a small cel highlight/accent so it reads as *authored*. A simple icon that is *clearly* its thing (a crisp flame, a solid shield) beats an ornate one that's ambiguous.
- **Cohesive *and at the same FIDELITY* as the hero art — no element is too small to break it.** Bars, buttons, orbs, pips, status/intent icons, the cost badge must share the **palette, outline weight, and shape-language** of the generated art AND its *level of finish*. This is the trap that bit this game: a flat geometric primitive drawn in code (a plain triangle for "attack", a 2-polygon flame, a bare gradient orb for mana) is *crisp* but reads as a **lower-quality, different hand** next to richly-painted cards/enemies — the owner called the whole code-drawn icon set "low quality and not cohesive." **Against fully-painted art, crude code/vector icons are an incohesion bug, not a clever mixed-method win.** For a high-fidelity painted title, the icons must be **generated as painted assets to the same fidelity** (a tiny flame/shield/mana-gem token from the same checkpoint), or — only if generation truly can't render them cleanly — styled in code to genuinely *match* the painted finish (rendered shading, not flat geometry). "It's just a little icon" is exactly how an otherwise-finished screen ends up with a ring of programmer-art tokens. (This **supersedes** the earlier "small icons → mixed-method code/vector for crispness" instinct: crispness is necessary but not sufficient — fidelity-cohesion outranks it whenever the surrounding art is painted.)
- **Legible — text must not collide with its own element.** A value drawn *on top of* its bar/badge — the classic **"24/30" HP number overlapping the HP-bar fill** — is a bug: move it beside/above, or give it a solid backing pill. Re-check at true on-screen size.
- **Detail matched to display size — AND mipmaps on, or everything downscaled looks grainy.** An asset authored ornate at high res and then **scaled down into a small UI slot aliases into noise/grain** (owner finding: the card-back + relic look grainy as ~44px thumbnails). Two independent causes, fix both: (1) **Technical — mipmaps.** A 768×1024 texture drawn at 44px with **no mipmaps** samples the full-res image and aliases. Godot's importer defaults `mipmaps/generate=false`; set it **`=true`** in each downscaled texture's `.png.import` (then re-import), AND set the canvas's texture filter to a **mipmap** variant (`CanvasItem.TEXTURE_FILTER_LINEAR_WITH_MIPMAPS`, e.g. `texture_filter = ...` on the drawing Node2D) — mipmaps only kick in when *both* are set. This one fix de-grains *every* downscaled draw (thumbnails, hand cards, sprites), so do it before blaming the art. (2) **Authoring — bold/simple source.** Even with mipmaps, a busy/noisy source thumbnails worse than a bold simple one; author small-display elements with **fewer, larger shapes + thick outlines**, or generate a deliberately low-detail variant. **Verify the DOWNSCALED result at true size, not the full-res PNG** — "the source looks crisp" ≠ "the 44px thumbnail looks crisp."
Treat **HP/health bars (player AND enemy), resource pips, and primary buttons (End Turn / confirm)** as **first-class styled elements the audit must name by default** — they are high-frequency, high-visibility, and the things most often left as raw primitives. "It's just a bar/button" is the rationalization that ships programmer-art.

**Hold UI micro-assets to the hero-art QUALITY bar, judged at TRUE SIZE — "wired" and "painted" do not mean "good."** Replacing a primitive with a generated icon is not the finish line: the generated icon can itself be low-quality and must be caught and re-rolled, not shipped because it's "art now." After wiring, audit **each micro-asset on its own merits at true display size**, by name (resource orbs, cost gems, rarity/element pips, pile/deck counts, status icons, intent icons, relics, buttons, bars):
- **Instant clarity** — nameable in under a second at true size? (Owner: the sword intent icon was "very low quality" — its detailed hilt turned to mush at ~18px.) Small slots need **bold, chunky, minimal-detail** subjects; fine detail aliases away.
- **Brightness / contrast / pop** — a token that should glow reads muddy or dark? (Owner: the generated "mana" came back a dark glossy sphere — finished but wrong.) Re-roll with the right value or post-tint.
- **Semantic colour** — the hue must match the meaning: mana = bright blue, fire = warm amber, ice = cool cyan, health = green, gold = amber. A dark/grey mana or off-hue element fails even with the right shape.
- **Legible numerals on badges** — numbers on cost gems, pile/deck counts and status stacks must be crisp and high-contrast at true size; a thin number on a small/busy/dark badge is a fail — solid backing + shadow + size up.
- **Re-roll the generator, don't ship the first roll** — small-icon generation is unreliable (multi-subject, wrong colour, ambiguous, dark); budget 2–3 regens per icon with single-subject + colour locks, and when a subject keeps fighting switch to a clearer **concrete object** (snowflake→ice-shard worked here). One subpar roll on each of ten tokens = a UI that reads cheap even though every piece is "done."
The failure mode this prevents: a screen of individually-"finished" tokens that are each *slightly* low-quality (a dark orb here, a mushy sword there, a faint count) so the whole UI reads cheap despite great hero art. Hold every token to the same bar as the cards and enemies.

**The audit is a LOOP, not a one-shot — re-render and re-audit after EVERY fix round.** Implementing the fixes is not the end; **screenshot the running game again and run the whole composited-game audit against the NEW frame.** Two reasons it's mandatory: (1) **code-right ≠ screen-right** — confirm each fix actually resolved the issue *in the composite* at true size (a restyled bar can still clip, a "centered" value can still sit a pixel off); (2) **fixes spawn new issues** — a repositioned element creates a fresh collision, a newly-styled panel exposes a primitive that was hidden behind the old one, a bolder icon now overpowers its neighbor. Keep looping **audit → fix → re-render → re-audit** until a full pass surfaces nothing new. This is the verification-before-completion / completeness-critic discipline applied to art: never call the art pass done off the screenshot you *fixed against* — call it done off a *clean re-audit* of the screenshot that came *after* the fixes.

### When the base model FIGHTS the style, swap the checkpoint — don't out-prompt it (2026-06-03, owner directive)
The single highest-leverage style lever is the **checkpoint**, not the prompt. The proven failure (deckbuilder-0001 probe, ~9 imp + ~9 card regens): the target was a flat, bold, 2D cartoon look (Dicey-Dungeons / Slay-the-Spire register), but the base checkpoint was **Juggernaut XL — a photoreal finetune**. No amount of `2D, flat, cel-shaded, hand-drawn` + anti-`3d/grit/photo` negatives fully removed its inherent **rendered-volume / gritty-texture / "looks 3D"** bias; each regen fixed one axis and slipped on another. **A photoreal base model will keep dragging representational subjects toward realism — that is a property of the weights, not the prompt.** When 3–4 honest style regens still read wrong *in the same direction* (too rendered / too realistic / too gritty), stop out-prompting and **swap to a checkpoint whose native output already IS the target style** (a flat-cartoon / illustration / anime SDXL finetune). One checkpoint swap did in a single generation what a dozen prompt iterations could not, and it also **solves cross-asset cohesion for free** — every asset inherits the model's one hand, so the whack-a-mole between creature and effect disappears.
- **How to source one (no token needed):** query the Civitai API for an SDXL checkpoint matching the style — `curl -s "https://civitai.com/api/v1/models?limit=20&types=Checkpoint&baseModels=SDXL%201.0&tag=cartoon&sort=Most%20Downloaded"` → parse `items[].name` + `modelVersions[0].files[0].downloadUrl`. Download anonymously (`curl -L --fail -o models/checkpoints/<name>.safetensors "<downloadUrl>"` — it 307-redirects to a B2 CDN with an auto-issued token, returns the single-file `.safetensors`). HuggingFace single-file mirrors also work anonymously, but most HF "cartoon SDXL" repos (e.g. John6666/*) are **diffusers-format** (separate `unet/`,`vae/` folders) which `CheckpointLoaderSimple` can't load — you need a **single-file** checkpoint. (deckbuilder-0001 locked on `cartoonArcadiaXL_v2`, Cartoon Arcadia XL, Civitai ver 179901.)
- **Gate the new checkpoint before the batch:** ComfyUI picks up a newly-dropped checkpoint without a restart (folder re-scan on mtime). Generate ONE transparent sprite and confirm two things: (a) the **look** is the target style, and (b) the **LayerDiffuse alpha-join is clean** (no haze/gray-interior matte) — a non-standard finetune *can* break the join. A clean flat-cartoon SDXL finetune kept the join clean here, but it is the real compat risk, so prove it on one asset. Record the chosen `checkpoint` in every recipe + `asset_pass`; it is now part of the locked visual system.
- **Style words still matter, but they now *reinforce* a willing model instead of *fighting* a hostile one** — keep `flat bold cartoon, clean thick outlines, vibrant flat cel shading, bold graphic shapes` in the positive and `anime, manga` in the negative if you want Western-not-anime; the checkpoint does the heavy lifting, the prompt just steers tone/subject.

### Spell / ability cards: caster-driven, world-anchored, POV — don't fight for pure abstract effects (2026-06-03, owner directive)
An **abstract spell-effect** ("forked chain lightning, no figures") is one of the worst subjects for an opaque SDXL card: the model's "fantasy card art" distribution is **scenes and characters**, so a figure-less effect collapses to the nearest in-distribution thing — a random landscape (forest / canyon / ruins), a caster it invents, a weak scattered web, or even literal *trees* from "branching." (deckbuilder-0001: ~9 chain-lightning regens cycled through all of these.) Don't burn a dozen gens fighting it. Instead, give the model a subject it renders reliably **and** that ties to the game:
- **Caster-driven:** show the game's protagonist *performing* the spell (a recurring, consistently-described duelist — same robe/palette every card — so the cast coheres like a TCG's hero). Juggernaut/most SDXL models render this eagerly and cleanly.
- **First-person POV (owner's lever, strong):** frame the card as the spell **leaving the caster's own hand** — "first person POV looking down the caster's outstretched hand, <effect> bursting from the palm." This reads instantly as the ability, is distinctive, and **strips the rendered-figure/background detail** so a character-card sits closer to a creature sprite's simplicity (helps cross-asset cohesion).
- **World-anchored:** set it in the game's actual environment (the duel chamber) rather than letting the model pick a random vista — directed scene > drifting scene, and it ties the card to the world bible.
- Still audit each card with **"could a player name this spell from the art?"** (the named-subject rule above): a caster holding *fire* on a card named Chain Lightning is still a miss — negative the wrong element, name the right one.

### Per-entity flow
For each entity you decide to make raster:
1. **Recipe** — compose the JSON recipe: `prompt` = `scaffold + this actor's subject`, where the subject is **this actor as a member of the world-bible family** (Step 0) — never a free-standing description; plus `negative`, `seed`, `sampler`, `steps`, `cfg`, `checkpoint`, optional `lora`, `layerdiffuse: true`, and `master_resolution` (see Resolution below). Keep `sampler`/`steps`/`cfg` identical across the game's sprites. A template that includes a `%lora%` node requires the recipe to set `lora`; for a profile with no LoRA, use a template that omits the `%lora%` token (an absent `lora` against a `%lora%` template fails loudly by design — that is attributable, not a bug).
   - **Proven defaults (feasibility gate):** use an **SDXL finetune** as `checkpoint` (e.g. Juggernaut XL, DreamShaper XL) — **not base `sd_xl_base_1.0`**, which renders flat/washed-out through LayerDiffuse. `sampler: "euler"`, `cfg: 7–8`, `steps: 20–25`. The LayerDiffuse templates bake the FG-RGBA-canonical settings (`SDXL, Conv Injection` + scheduler `normal`) — those are the *working* settings; `Attention Injection`/`karras` produced mud in testing. `master_resolution`: 1024 for a hero/large actor, 512 for a minor prop (always downscale-from-master).
   - **Sprite-gen tuning levers (round-3, opt-in).** Owner chose to expose these; the round-3 probe found sprite tuning *low-ROI* vs. backgrounds, so reach for backgrounds/sizing first. Two levers exist: (1) **scheduler** — recipes may set `"scheduler":"karras"` (default `"normal"`); note the prior feasibility finding that `karras`+`Attention Injection` "produced mud", so confirm by eye before adopting (still UNPROVEN — no GPU probe ran on karras). (2) **refine pass** — set `"refine":true` to route to the `sdxl-layerdiffuse-refine` template (a `LatentUpscaleBy` 1.5× + a denoise-0.45 second pass → ~1536² master). **GPU-VERIFIED CLEAN (2026-06-02):** the LayerDiffuse alpha-join survives the upscaled latent — a refined 1024→1536 spirit kept fully transparent corners (alpha 0.00), the same transparent fraction (0.61 vs the 1024 original's 0.62), and equivalent edge quality, no matte halo. But the quality gain is **marginal** (the round-3 reframe: sprites were already fine), so treat refine as a crispness / high-DPI-master lever, not a fix for a "bad" sprite, and weigh it against churn to any store assets derived from the sprite. `refine` + `lora` together is not supported this round.
2. **Generate** — `node tools/comfy.mjs gen <id> <name> '<recipe-json>'` → writes `games/<id>/art/<name>.png` (RGBA). On a graph/unreachable error it fails loudly — fix the *infra/recipe*, do not fake the file.
3. **Import** — run the headless import pass so Godot makes the `.png.import` sidecar + cached texture **before** re-validation:
   ```
   godot --headless --path games/<id>/ --import
   ```
   - **Re-run `--import` after EVERY regeneration that overwrites an existing PNG, or Godot serves a STALE cached texture.** (deckbuilder-0001 probe: an imp regenerated to a new look still rendered as the *old* imp in the capture, because the cached `.ctex` wasn't invalidated — meanwhile a different asth-changed file did re-import, masking the bug.) Overwriting the `.png` in place does not reliably trigger re-import on a headless `--script`/screenshot run; the explicit `--import` pass does. Treat "regenerate art → `--import` → render/validate" as one atomic sequence — never render straight after a `gen` that overwrote a file.
4. **Configure mobile import settings** (see Resolution & mobile density) — edit the generated `games/<id>/art/<name>.png.import` so it is mobile-grade, then re-run `--import`.
5. **Rewire** — identical to the SVG swap below: prefer `draw_texture*`-in-place for the builder's default single-`_draw()` games, else pooled `Sprite2D`/`TextureRect`; place at the primitive's original footprint; **remove/guard** the matching primitive (no double-draw); **game logic untouched**.
   - **Remove stale glow-halos sized for the old primitive.** When you replace a small primitive blob with a larger raster sprite, any `draw_circle` glow-halo that was sized for the old primitive footprint now sits behind the bigger sprite and reads as an odd disc artifact. Delete it along with the primitive draw call.
   - **Variable-width actors: tile exact-fit units, never stretch.** When a hazard or actor has a runtime-variable width (e.g. a bar whose length changes), fill it by tiling N = `round(w / unit_size)` sprite units that each fit exactly, so visual width matches collision width and the art stays undistorted. Stretching a fixed-aspect sprite to a variable width distorts it and misaligns the visual and collision bounds.
   - **Tab-indented GDScript and exact-match edits are fragile.** The builder's GDScript uses tab indentation; the Edit tool's exact-match frequently fails on tab-vs-space ambiguity. A reliable workaround: use PowerShell to read the file into a lines array, reconstruct around the target range, and write back. `Get-Content`/`Set-Content` preserve tab indentation across the round-trip. For a multi-line splice (0-based indices `$a..$b`):
     ```powershell
     $lines = Get-Content path/Main.gd
     # replace the inclusive line range [a..b] (0-based) with $new (a string array)
     $out = $lines[0..($a-1)] + $new + $lines[($b+1)..($lines.Count-1)]
     Set-Content path/Main.gd $out
     ```
     For a single-line replacement, `$lines[$i] = 'new line'` is enough, then `Set-Content`. Prefer this over repeated Edit retries when a GDScript splice isn't landing.

### Backgrounds & composition (round-3 — the real art-quality lever)

Round-3 playtests proved the sprites were already fine; "terrible" came from **flat primitive backgrounds + heroes too small**, not sprite quality. So this method now **generates environment art and sizes the hero deliberately** — this is where art quality actually lives.

> **GPU-VALIDATED at Phase 2 (2026-06-02).** Background generation and the refine pass were originally codified from reasoning; they have now been run on the RTX 5080 against creature-0001 (painterly) and crosser-0001 (pixel). Both produced strong, on-theme backgrounds that were in-frame-verified (clean composite, no double-draw; refine alpha-join stayed clean — see above). What remains pending is the **owner's aesthetic A/B sign-off**, not the technical mechanism — these defaults are now trustworthy to *start* from.

- **Background = full-frame OPAQUE image (no LayerDiffuse).** Use the plain `sdxl` template: recipe with `layerdiffuse:false`, **no** alpha, and explicit non-square `width`/`height` matching the game's aspect (e.g. `1280×768` landscape, `768×1280` portrait — `comfy.mjs` now honors distinct `width`/`height`). The prompt expresses `concept.theme.setting` as a *scene/environment* ("a cozy autumn-woodland glade, soft depth, storybook") — the same world the sprites inhabit. Draw it as the **bottom layer** (root `_draw()` background, or a full-rect `Sprite2D` at `z_index` below all actors) replacing the flat primitive band/void. Record it as a `reskinned` background entity with `origin:"raster"`.
- **This supersedes the old "background left primitive is an M1.7-deferred gap" default** (see the immediate-mode §c note) — a themed background is now **in scope and expected** for a representational title. Leaving it primitive is now the exception you must justify, not the default.
- **Size the hero for the frame.** A detailed sprite floating tiny on a large field reads cheap. Place the hero so it occupies a **prominent share of its play area** and is clearly larger than / distinct from hazards (round-3: the crosser cyan hero was tiny next to red hazards). Sizing is a *runtime scale at wire time* (the `Sprite2D` scale / `draw_texture_rect` dest size), not a generation parameter — set it so the hero reads as the subject. Record the intent in `asset_pass.notes`. (Phase-2: creature `SPIRIT_DRAW` 110→150, crosser `HERO_DRAW` 64→80 — visual-only bumps, hitboxes untouched; both verified in-frame.)
- **Pixel-art backgrounds: the opaque `sdxl` template has NO LoRA node (Phase-2 finding).** A pixel-art *sprite* uses the `sdxl-layerdiffuse-lora` template to apply `pixel-art-xl`; but the **opaque** background template (`sdxl`, `layerdiffuse:false`) carries no `%lora%`, so a pixel LoRA can't drive the background. Get the pixel look from **prompt terms** ("pixel art, 8-bit, NES, chunky pixels") + the project's **NEAREST `texture_filter`** (set on the Main node, inherited by `draw_texture_rect` at draw time). This worked well in Phase-2 (crosser bg) — record the LoRA-absence as a recipe `note` so it reads as a known limitation, not a miss.
- **Full-PLAYFIELD games: a background image only fixes the MARGINS (Phase-2 finding).** For a title whose playfield fills the screen (e.g. a lane-crosser whose flat lane bands ARE most of the frame), a full-frame background only enriches the visible margins around the field — the flat playfield itself still reads flat against the rich backdrop. Fully fixing that needs **lane/surface TILE textures** (a different technique: `draw_texture_rect` tiled per lane, with seamless tiles), which also risks the lane-color *readability* the gameplay depends on. Don't speculatively retexture a functional playfield — ship the margin/sizing win, record the lane-tile opportunity as a finding, and let the owner direct whether to pursue it.

### Text & chrome over full-bleed art — the full-bleed card pattern (2026-06-02)
When a card/panel uses a **full-bleed painted illustration** (opaque art filling the whole rect, no authored frame) and code draws the gameplay text on top, the art is the *background* of the widget by design — **the generated PNG carries no border, panel, or text** (those would bake in wrong values and fight legibility). All chrome is **code-drawn at runtime**, so the same renderer works for every card. The pattern:
- **A defined semi-transparent SOLID panel behind each text block — not a gradient scrim alone.** A gradient scrim fades to transparent and so fails over a *bright* patch of art (the owner's finding: "the text is hard to read"). A solid panel (`draw_rect` with `Color(~0.05, 0.04, 0.11, ~0.80)`) gives **consistent** contrast regardless of what the diffusion model painted underneath. Feather just its leading edge with a short gradient quad so it doesn't hard-cut into the painting. Use one panel for the bottom effect block and one for the top name.
- **Drop-shadowed text** on top (draw the string once in near-black at +1px, then in the real colour) for an extra contrast margin.
- **Element border + cost badge + element pip** stay code-drawn over everything (gameplay-critical at-a-glance identity), drawn *last*.
- **Gradient-only scrims are a smell here** — they read fine in a mock but fail on real art's bright regions. Prove legibility on a *real* generation (a throwaway probe scene that composites art + the actual chrome) before committing the batch, and judge it at true on-screen size, not just zoomed.
- **Scale text with the widget, not absolute px.** A card drawn bigger (reward cards, a 2× detail view) must get proportionally bigger text — derive font sizes + chrome geometry from `rect.size` (e.g. `s = rect.size.y / base_h`), don't hard-code px, or big cards show tiny text.
- **Chrome-layout QA — audit the COMPOSITED result for collisions/clipping, not just the art.** The owner's audit covers the rendered widget too: **no element overlaps another** (the classic: a centered name running under the top-left cost badge — center the name in the space *right* of the badge), text **fits inside its panel** and is not **clipped** by the card edge, and the cost/border/pip stay clear of the text. Eyeball every state (affordable / unaffordable / selected) and the largest size. A layout collision is an attributable chrome bug, exactly like a subject/tone drift.
- Immediate-mode gradient quad helper (per-vertex colours): `draw_polygon(corners, [c_top,c_top,c_bot,c_bot])`.

### Mixed-method honesty (inverted boundary)
The SVG method flags when representational art needs raster. Raster **inverts** it: if an actor is genuinely better as crisp vector or a primitive — a UI/HP bar, a geometric pickup, a HUD frame — you are **allowed to leave it `svg` or `primitive`**, and you must **say so**. A run can be mixed-method. Record per-entity which is `raster` vs `svg` vs `primitive` (`reskinned`/`left_primitive` + `assets[].origin`) so a partial raster pass is a legible choice, not a silent gap.

**When an actor fights the medium, mixed-method is the right call — not a cop-out.** If a sprite is tiny or effect-like (e.g. a ~14px glowing pickup), SDXL will struggle to render it coherently across multiple retries; a procedural primitive (a glow-mote `draw_circle`) is the higher-quality result. Vary in this order: subject phrase first, then `master_resolution` (1024 → 512 → 256), then `cfg` by ±1 — if it still fights you, that actor fights the medium. After exhausting 3–4 prompt/param variations without a clean output, leave that actor primitive and record the finding. Trying harder on a bad-fit actor burns time without improving the game.

### Resolution & mobile density (recovering what SVG gave for free)
SVG covered every Android density bucket (mdpi → xxxhdpi) from one file. Raster forfeits that, so replace it deliberately or the output is **not** app-store-ready:
- **Generate high-res masters.** Size each sprite's `master_resolution` for the **largest** density bucket it occupies on screen — a generous power-of-two master (e.g. **512²** for a minor prop, **1024²** for a hero actor). Always **downscale from the master**, never upscale, so it stays crisp on xxxhdpi.
- **Set mobile import settings explicitly.** In each `games/<id>/art/<name>.png.import`, set `mipmaps/generate=true` (clean minification), a linear `filter`, and a 2D-mobile-appropriate compression (lossless for small sprite counts; VRAM-compressed if the title grows). Do **not** accept Godot's editor defaults silently. Record the choice in the recipe's `import_settings` (`{ mipmaps, filter, compression }`).
- **Pixel-art crispness: downscale the master on import, then draw with `NEAREST`.** For pixel-art style profiles, set `process/size_limit` in the `.png.import` to a small target size (e.g. 64 or 128) so Godot resamples the 1024 master down cleanly at import time. Then draw with `TEXTURE_FILTER_NEAREST` (mipmaps off) to keep chunky pixels crisp. Do **not** draw a raw 1024 master tiny with nearest filtering — the alias noise is severe. The size_limit resample is the clean first step; nearest filtering then locks the chunky pixel look.
- **Footprint mapping unchanged.** The sprite scales to the primitive's on-screen footprint at runtime; the master being larger than the footprint is *correct* (high-DPI headroom), not waste to "fix" by shrinking the master.

Texture **atlasing** and whole-title APK **size budgeting** are **M2** packaging concerns — out of scope here. Generating proper masters now is what makes that later step possible.

### Content & IP safety
SDXL can emit outputs resembling trademarked/copyrighted characters — an app-store rejection and legal risk. So:
- Put IP guards in **every** recipe's `negative` prompt: `"logo, watermark, text, trademarked character, brand, celebrity likeness"`.
- Keep prompts to **generic descriptors** — never name a franchise, studio, or character.
- The **human A/B** is the safety review: the owner confirms nothing looks like protected IP **before** `styled`. (Model/output licensing is settled in the research note — SDXL Community License, outputs owned for commercial use under the revenue threshold; re-verify the threshold at ship time.)

### Determinism (state this plainly)
The committed **PNG is canonical** — a fresh clone runs identically. `recipes[]` is **provenance, not a bit-exact regeneration guarantee**: GPU matmul + RNG are non-deterministic, so the same recipe regenerates a *close* image, never the same pixels. "I have the seed" ≠ "I can reproduce the art."

## Recording the pass

1. Flip the re-skinned `assets[]` entries (arrays replace wholesale — pass the **full** array, re-skinned entries as `origin:"svg"`, untouched ones as-is):
   ```
   node tools/manifest.mjs merge <id> "{\"assets\": [ {\"type\":\"sprite\",\"name\":\"player\",\"source\":\"art/player.svg\",\"origin\":\"svg\"}, ... ]}"
   ```
2. Write the `asset_pass` block (record the Step 0 system verbatim):
   ```
   node tools/manifest.mjs merge <id> "{\"asset_pass\": {\"method\":\"svg\",\"visual_system\":{\"world_bible\":\"...\",\"palette\":[...],\"stroke\":\"...\",\"form\":\"...\",\"shading\":\"...\",\"scale\":\"...\"},\"reskinned\":[...],\"left_primitive\":[...],\"art_path\":\"games/<id>/art/\",\"notes\":\"...\"}}"
   ```
3. Validate the manifest (still `playable` at this point):
   ```
   node tools/manifest.mjs validate <id>
   ```
   Expected: `<id> OK`.

**For the `raster` method**, the `asset_pass` you merge in step 2 additionally carries the style profile, the shared scaffold, and one recipe per generated PNG (arrays replace wholesale — pass the full `recipes[]`):
```
node tools/manifest.mjs merge <id> "{\"asset_pass\": {\"method\":\"raster\",\"visual_system\":{\"world_bible\":\"...\",\"palette\":[...],\"form\":\"...\",\"shading\":\"...\",\"scale\":\"...\",\"prompt_scaffold\":\"...\",\"style\":{\"checkpoint\":\"...\",\"loras\":[...],\"style_prompt\":\"...\"}},\"reskinned\":[...],\"left_primitive\":[...],\"art_path\":\"games/<id>/art/\",\"notes\":\"...\",\"recipes\":[{\"name\":\"hero\",\"checkpoint\":\"...\",\"prompt\":\"...\",\"negative\":\"...\",\"seed\":123,\"sampler\":\"...\",\"steps\":30,\"cfg\":6.5,\"master_resolution\":1024,\"layerdiffuse\":true,\"lora\":\"...\",\"import_settings\":{\"mipmaps\":true,\"filter\":\"linear\",\"compression\":\"lossless\"}}]}}"
```
Flip each raster entity's `assets[]` entry to `origin:"raster"` (e.g. `{"type":"sprite","name":"hero","source":"art/hero.png","origin":"raster"}`); leave `svg`/`primitive` entities as they are.

## Hand off to the validator

Do **not** set `styled` yourself. Hand off to `validator`, which re-runs the same gates on the rewired game (headless import + run clean; `selftest.gd` still `SELFTEST OK` if present; human A/B playtest) and advances `playable → styled` on success, or records legible `issues` (attributed almost always to `asset`) and stops on failure.

## Notes
- The import pass (`--import`) must run before the validator's headless run, or `load("res://art/...svg")` returns null at runtime.
- If a re-skin makes the game look *worse* or incoherent, that is a finding about the **visual system** (Step 0), not the individual files — fix the system and re-derive, don't patch one SVG.
